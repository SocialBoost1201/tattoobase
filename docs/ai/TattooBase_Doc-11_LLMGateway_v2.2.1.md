# TattooBase Doc-11 LLMGateway
Version: v2.2.1
Last Updated: 2026-02-22
Owner: AI Engineering

Change Summary:
- Gateway抽象化とプロンプト管理層の定義
- レスポンス検証と出力サニタイズの必須化
- コスト・セキュリティ制御の明文化
- AIActivityLogとの連携（監査可能性必須）

## 1. 原則と目的
LLM（大規模言語モデル）からの出力は本質的に非決定的であるため、そのままビジネスロジックに直結させてはならない。
外部LLMはユーザー体験の向上とStudioへの提案・分析補助としてのみ利用する。
コアロジック（予約ステートマシン、決済、Risk Engineによるハードルールによる制御など）はLLMに絶対に依存してはならず、LLMがダウンしても中核業務は必ず継続できる設計でなければならない（失敗時フォールバックの徹底）。

## 2. アーキテクチャ構成

### 2.1 Gateway抽象化
- `LLMGateway` インターフェースを定義し、アプリケーションロジック（`AIService` 等）は特定の外部プロバイダ（OpenAI, Anthropic 等）に直接依存しない。
- 複数のLLMプロバイダをエンドポイント単位、あるいはユースケース単位で切り替える機能（モデル差替可能設計）を持つ。

### 2.2 プロンプト管理層
- システムプロンプトおよびユーザープロンプトのテンプレートは、ソースコード上にハードコードせず、モジュールとして分離して管理する。
- バージョン管理を導入し、「どのバージョンのプロンプトでどういう出力を得たか」を後から追跡可能にする。

### 2.3 レスポンス検証層
- LLMからの出力は必ず不安定（ハルシネーションやフォーマット不正）である前提に立つ。
- JSON Schema等による厳格な構造検証(Parsing & Validation)を行い、スキーマ検証・パース失敗時は即座にエラーとしてフォールバック処理に移行する。

## 3. セキュリティとデータ保護

### 3.1 PII送信最小化とセキュリティ制御
- 顧客の個人情報（氏名、電話番号、メールアドレス等）およびKYC情報（身分証データ）は、いかなる場合も外部LLMに送信してはならない。
- システム内で匿名化（User IDによる置き換え等）を前処理で行い、推論完了後に表示のために後処理で結合する。

### 3.2 出力サニタイズ
- LLMの出力をそのままDBやUIに反映してはならない。
- 出力されたテキストデータには必ずサニタイズ処理（XSS対策、禁止語句・暴力的表現のフィルタリング、不適切なHTMLタグ除去等）を施す。

## 4. 運用・管理設計

### 4.1 失敗時フォールバック
- LLMプロバイダのAPIエラー（500系エラー）、応答タイムアウト、またはレスポンス検証失敗時は、以下のフォールバックを実行し、処理をブロックしない。
  - カスタマー向け機能: エラーを見せず、標準的な静的メッセージや既存のルールエンジンのみに依存した結果を返す。
  - Studio向け機能: 「AIによるサマリー・提案は現在利用できません」と表示し、人間の手動操作を促す。

### 4.2 コスト制御
- リクエストごとのトークン使用量（Prompt Tokens / Completion Tokens）を計測する。
- スタジオ単位・エンドポイント単位で1日/1ヶ月の利用枠（Quota）を設け、意図せぬ大量リクエストによる請求急増（Billed Exhaustion）を防ぐ。
- ハードリミット到達時は自動でフォールバック機構（エラー/静的応答）に切り替える。

### 4.3 監査ログ (AIActivityLog)
- 全てのLLM経由のやり取りは `AIActivityLog` サブシステムに記録する（通常のシステム `AuditLog` とは論理的に分離するが、トレーサビリティは同様とする）。
- 実装・記録必須項目：
  - `requestId` (トランザクション全体を追跡するID)
  - `actorId` (要求元のUserId/StudioId)
  - `promptVersion`
  - `sanitizedInputSummary` (匿名化済み入力要約)
  - `sanitizedOutput`
  - `tokensUsed`
  - `latencyMs`
